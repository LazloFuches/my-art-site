# Robert Egert Studio - Website Documentation

Last updated: 2026. This document describes how the robertegert.com website
is built, hosted, and maintained. It lives in the project folder (not on the
public website) as a reference for future updates.

================================================================
PART 1 - PLAIN-LANGUAGE OVERVIEW (for any reader)
================================================================

WHAT THIS SITE IS
-----------------
robertegert.com is a "static" website: an archive of art projects, a
biography, an ephemera section, and a contact form. "Static" means the
website is a set of pre-built files that are simply handed to visitors -
there is no database and no software running on a server answering each
visit. This makes it fast, secure, and nearly maintenance-free.

HOW UPDATES REACH THE LIVE SITE
-------------------------------
Making a change involves four places working together:

1. YOUR COMPUTER - where you edit the source files (in the "my-art-site"
   folder) using the VS Code editor.
2. ELEVENTY - a program on your computer that turns the source files into
   the finished website. Run with a command; you don't edit it.
3. GITHUB - an online storage service that keeps every version of the
   project. You "push" changes here.
4. CLOUDFLARE - watches GitHub, rebuilds the finished site whenever you push,
   and serves it to the public at robertegert.com.

The cycle: edit files on your computer, preview locally, save the change to
GitHub, and Cloudflare automatically rebuilds and publishes. A change is
usually live about a minute after you push.

THE ACCOUNTS INVOLVED
---------------------
- GitHub - stores the project code and images. (Account: personal email.)
- Cloudflare - hosts the site and manages the domain's DNS.
- Formspree - receives contact-form submissions and emails them to you.
- The domain robertegert.com - registered via Tucows/Siteground; its DNS is
  now managed by Cloudflare.

Passwords and keys are NOT in this document by design. Keep them in a
password manager.

PART 2 - TECHNICAL REFERENCE
================================================================

TECHNOLOGY STACK
----------------
- Eleventy (11ty) v3.x - static site generator. Reads source files, outputs
  finished HTML into the _site folder.
- Nunjucks (.njk files) - templating language. Files contain HTML plus
  placeholders and logic that Eleventy fills in.
- Node.js / npm - runs Eleventy and manages installed packages.
- GLightbox - small JavaScript library for the click-to-enlarge image
  lightbox on project and ephemera pages.
- Git - version control; syncs the local project to GitHub.
- Cloudflare Pages - builds and hosts the site.
- Formspree - third-party form handler for the contact page.

FOLDER STRUCTURE (inside my-art-site/)
--------------------------------------
  src/                     All source files (this is what you edit)
    _includes/
      base.njk             Master template: header, nav, footer, head,
                           schema markup. Every page uses this.
      project.njk          Template for an individual project page.
    _data/
      projects.json        The list of all projects (title, slug, year,
                           studio, thumbnail path, work count). Drives the
                           homepage grid.
    projects/
      <slug>.njk           One file per project, containing its image list
                           and captions in the front matter.
      index.njk            (Legacy) projects index.
    assets/
      css/style.css        All site styling.
      images/<slug>/       Images for each project, plus thumb.jpg.
      images/ephemera/     All ephemera images and PDF cover images.
      pdfs/                Downloadable PDF catalogues and documents.
      glightbox.min.js     Lightbox library.
    index.njk              Homepage (project grid).
    bio.njk                Biography page (essay + exhibition history).
    ephemera.njk           Ephemera page.
    contact.njk            Contact form page.
    sitemap.njk            Generates sitemap.xml for search engines.
    robots.txt             Tells search engines where the sitemap is.
  _site/                   The FINISHED site, generated by Eleventy. Never
                           edit by hand; it is rebuilt every time. Not stored
                           in GitHub.
  .eleventy.js             Eleventy configuration.
  package.json             Lists installed tools.
  node_modules/            Installed tool code. Not stored in GitHub.
  .gitignore               Tells Git to ignore node_modules, _site, .DS_Store.
  ADDING-IMAGES.txt        Step-by-step guide for adding an image.
  TO-DO.txt                Running task list.
  DOCUMENTATION.md         This file.

THE PUBLISHING WORKFLOW
-----------------------
Two terminals in VS Code are convenient: one running the live preview, one
for commands.

1. Start the local preview:
     npm run start
   Then open http://localhost:8080 in a browser. It auto-refreshes as you
   edit. Press Ctrl+C to stop it (click in the terminal first).

2. Make your edits to files in src/. The preview updates automatically.

3. When satisfied, publish:
     git add .
     git commit -m "short description of the change"
     git push
   (These can be chained with && between them.)

4. Cloudflare detects the push and rebuilds automatically. The change is
   live at robertegert.com in about a minute. Check the Cloudflare dashboard
   if you want to watch the build.

COMMON TASKS
------------
ADD A NEW PROJECT:
  1. Create a folder src/assets/images/<slug>/ and add web-sized images
     named 01.jpg, 02.jpg, etc., plus a thumb.jpg.
  2. Create src/projects/<slug>.njk with the project's title, year, layout,
     and an "images:" list (copy an existing project file as a template).
  3. Add the project to src/_data/projects.json (title, slug, year, studio,
     thumbnail path, count).
  4. Preview, then commit and push.

ADD AN IMAGE TO AN EXISTING PROJECT:
  See ADDING-IMAGES.txt for the detailed version. In brief: add the image
  file to the project's image folder, copy the commented template block at
  the bottom of the project's .njk file, remove the # symbols, fill in the
  values, save, commit, push.

ADD EPHEMERA:
  Add the image (and a PDF + cover image if applicable) to
  src/assets/images/ephemera/ (and src/assets/pdfs/). Add a new entry block
  to src/ephemera.njk in the correct chronological position (newest at top).

EDIT TEXT (homepage line, bio, etc.):
  Edit the relevant .njk file directly, save, commit, push.

CHANGE THE NAVIGATION:
  Edit the nav section in src/_includes/base.njk.

IMAGE PREPARATION STANDARDS
---------------------------
- Format: JPEG, exported via Photoshop "Save for Web," quality ~80.
- Size: longest dimension 2000px for standard/large works; smaller
  (600-1000px) for physically small works so on-screen scale stays honest.
- Thumbnails: named thumb.jpg, ~800px longest dimension.
- Filenames: lowercase, no spaces, use hyphens. Extensions must be lowercase
  (.jpg not .JPG) - see gotchas below.
- Keep original/high-res files elsewhere (e.g. Dropbox); only web-optimized
  versions go in the project.

KNOWN QUIRKS AND GOTCHAS
------------------------
- CASE-SENSITIVE FILENAMES: Your Mac treats "photo.JPG" and "photo.jpg" as
  the same file, but Cloudflare's servers do not. If an image works locally
  but shows a broken link on the live site, the file extension case likely
  doesn't match what the HTML references. Fix: rename files to lowercase and
  ensure Git records the change.
- YAML INDENTATION: In the front matter of .njk files (the part between the
  --- lines), indentation matters. Each image entry's "- src:" must have
  exactly two leading spaces, and the fields below it four. A wrong indent
  breaks the page build.
- SPECIAL CHARACTERS IN TITLES: A "#" in a title can be misread. If a title
  contains one, wrap the title value in quotes in the .njk file.
- EDITOR SAVE: Always confirm a file saved (the dot on the VS Code tab means
  unsaved). If a page looks wrong, verify the file's contents in the terminal
  with:  cat <path-to-file>
- TWO TERMINALS: The preview command (npm run start) occupies a terminal.
  Use a second terminal for git commands, or stop the preview first.

DOMAIN AND DNS
--------------
- Domain: robertegert.com (registered via Tucows/Siteground).
- Nameservers point to Cloudflare, so Cloudflare manages all DNS.
- The site is connected in Cloudflare Pages under "Custom domains" for both
  robertegert.com and www.robertegert.com. HTTPS/SSL is automatic.
- Note: some stale DNS records from the old Siteground site may still exist
  in the Cloudflare DNS zone; they are inert and can be cleaned up.

SEARCH / SEO
------------
- Each page has a title tag and a meta description (default provided in
  base.njk; custom ones set per page in front matter).
- schema.org Person markup in base.njk identifies Robert Egert as an artist
  and links the Instagram profile (helps distinguish from "Roger Ebert").
- sitemap.xml is generated automatically and submitted to Google Search
  Console. robots.txt points to it.

TROUBLESHOOTING
---------------
- Page looks broken after an edit: check YAML indentation and that the file
  saved. Run "npm run build" and read any error message.
- Image broken on live site but fine locally: check filename extension case.
- Change didn't appear live: confirm "git push" succeeded and the Cloudflare
  build finished (check the Cloudflare dashboard).
- Live site broken by a bad change: Cloudflare keeps previous deployments and
  can roll back to a working one from its dashboard while you fix the source.
- Contact form not delivering: check the Formspree dashboard; submissions
  appear there even if email notification is delayed.
